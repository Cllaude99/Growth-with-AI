# Array & Linked List - 복습 노트

> 학습일: 2026-03-09

## Level 1: 기본 이해

### 주요 질문
- Array가 뭔지 본인의 말로 설명해볼 수 있나요?
- Array가 연속된 메모리에 데이터를 저장하기 때문에 얻을 수 있는 가장 큰 장점은?
- 특정 인덱스의 요소에 접근할 때 O(1)인 이유를 메모리 주소 계산 관점에서 설명해볼 수 있나요?
- Linked List는 Array와 어떻게 다른 구조인가요?
- 물리적으로 흩어져 있는 노드들을 어떻게 연결해서 순서를 유지하나요?
- Linked List에서 중간에 새로운 노드를 삽입할 때의 시간 복잡도는?
- 삽입할 위치의 노드를 이미 알고 있다면 삽입 자체의 시간 복잡도는?
- (검증) 정렬된 데이터에서 특정 값을 빠르게 찾아야 한다면 Array와 Linked List 중 어떤 걸 선택하겠고 그 이유는?

### 다시 공부할 내용
- **정렬된 데이터에서의 탐색**: 인덱스를 아는 접근(O(1))과 값을 기준으로 찾는 탐색을 구분하기. 정렬된 Array에서는 binary search로 O(log n)에 탐색 가능하지만, Linked List는 중간 인덱스 접근이 O(n)이라 binary search가 비효율적.

## Level 2: 트레이드오프

### 주요 질문
- 읽기가 많은 상황과 삽입/삭제가 많은 상황 각각에서 어떤 자료구조를 선택하겠고, 그 이유는?
- 삽입/삭제가 빈번하더라도 데이터 크기가 작고 순회도 자주 해야 하는 상황이라면, 여전히 Linked List가 유리한가요?
- Array와 Linked List 중 선택할 때 시간 복잡도 외에 어떤 요소들을 함께 고려해야 하나요?
- (검증) 수만 건의 요청을 앞에서 빼고 뒤에서 넣는 대기열을 구현한다면 어떤 걸 선택하겠어요?

### 다시 공부할 내용
- **데이터 크기가 작을 때의 판단**: 시간 복잡도상 Linked List가 유리해 보여도, 데이터가 작으면 Array의 cache hit rate가 높아 실제 성능은 Array가 더 빠를 수 있음. 삽입/삭제가 빈번해도 shift 비용이 무시할 만큼 작기 때문.

## Level 3: 심화

### 주요 질문
- Array가 꽉 찼을 때 2배씩 늘리면 append 연산이 amortized O(1)이 되는 이유는?
- 크기 1인 Array에서 시작해 8개 원소를 넣을 때까지 복사가 발생하는 시점과 각각의 복사 비용은?
- 2배가 아니라 매번 1칸씩만 늘린다면 총 복사 비용이 어떻게 달라지나요?
- Singly Linked List에서 특정 노드 삭제 시 이전 노드를 알아야 하는 문제를 어떻게 해결할 수 있나요?
- Circular Linked List는 어떤 구조이고 어떤 상황에서 유용한가요?
- (검증) Dynamic Array에서 절반이 비면 바로 절반으로 줄이는 게 좋을까요, 다른 전략이 나을까요?
- (재검증) Doubly Linked List에서 특정 노드를 삭제할 때 Singly Linked List와 비교해서 어떤 단계들이 필요하고 왜 더 효율적인가요?

### 다시 공부할 내용
- **Amortized O(1) 분석**: 2배 확장 시 총 복사 비용(1+2+4+...+n/2)을 전체 연산 수 n으로 나누면 상수가 된다는 원리. "총 비용 / 연산 수 = 평균 상수"라는 개념을 확실히 익히기.
- **Shrinking 전략과 thrashing**: 절반이 비면 바로 축소하면 경계에서 삽입/삭제가 반복될 때 확장-축소가 계속 일어남(thrashing). 1/4이 남을 때 절반으로 줄이면 축소 후에도 여유 공간이 있어 thrashing 방지.

## Level 4: 시스템 설계

### 주요 질문
- LRU Cache를 get O(1), put O(1)으로 구현하려면 어떤 자료구조를 사용해서 어떻게 설계할 수 있나요?
- key를 O(1)에 조회할 수 있는 자료구조는?
- HashMap과 Doubly Linked List를 조합하면 각각 어떤 역할을 담당하나요?
- 어떤 key를 get해서 사용했을 때 Doubly Linked List에서는 어떤 동작이 일어나야 하나요?
- (검증) 이 LRU Cache를 여러 thread가 동시에 접근하는 환경에서 사용한다면 어떤 문제가 발생하나요?

### 다시 공부할 내용
- **LRU Cache에서 두 자료구조가 필요한 이유**:
  - HashMap: URL 같은 key로 원하는 항목을 O(1)에 찾기. 순서 정보는 없음.
  - Doubly Linked List: 사용 순서 관리. 사용된 항목을 맨 앞으로 이동, 가장 오래된 항목은 맨 뒤에 위치 → 캐시가 꽉 차면 맨 뒤 노드를 O(1)에 제거.
  - 둘 중 하나만으로는 "빠른 조회"와 "순서 관리"를 동시에 O(1)로 할 수 없기 때문에 조합이 필요.
- **왜 Singly가 아닌 Doubly Linked List인가?**: 노드를 맨 앞으로 옮기려면 현재 위치에서 "빼내기" + 맨 앞에 "넣기"가 필요. 빼낼 때 이전 노드와 다음 노드를 연결해야 하는데, Singly Linked List는 이전 노드를 찾으려면 처음부터 순회해야 해서 O(n). Doubly Linked List는 이전 포인터가 있으므로 O(1)에 빼낼 수 있음.
- **Thread-safety**: 여러 thread가 동시에 접근하면 race condition 발생. Mutex(Lock)를 사용해 한 번에 하나의 thread만 접근하도록 보호.
