# Message Queue

## 메타데이터
- **카테고리**: 시스템 설계
- **난이도**: 중급
- **우선순위**: 중간
- **선수과목**: 없음
- **예상 시간**: 30-45분

## Level 1: 기본 이해

### 핵심 질문
- Message queue가 무엇인지 설명하고, 동기 통신 대신 비동기 통신을 사용할 때의 장점은 무엇인가요?

### 꼬리 질문
- Message queue를 사용하면 시스템의 어떤 부분이 decoupling 되나요? 이것이 왜 중요한가요?
- 비동기 처리의 단점은 무엇인가요? 사용자 경험 측면에서 어떤 영향이 있나요?

### 예상 이해 수준
Message queue가 producer와 consumer 사이에서 메시지를 버퍼링해 비동기 통신을 가능하게 하는 미들웨어임을 설명할 수 있어야 합니다. 비동기 통신의 장점으로 서비스 간 결합도 감소, 트래픽 급증 시 버퍼 역할, 처리 속도가 다른 컴포넌트 간 조율, 장애 격리 등을 설명할 수 있어야 합니다. 응답 시간 지연과 복잡성 증가라는 단점도 인식할 수 있어야 합니다.

## Level 2: 트레이드오프

### 핵심 질문
- Point-to-point(Queue) 모델과 Publish/Subscribe 모델의 차이점은 무엇인가요? 어떤 상황에서 각각을 선택하나요?

### 꼬리 질문
- 하나의 이벤트를 이메일 발송, 푸시 알림, 데이터 분석 세 가지 시스템이 모두 처리해야 한다면 어떤 모델을 선택하나요?
- Pub/Sub 모델에서 특정 consumer가 일시적으로 다운된 경우 메시지 유실을 어떻게 방지하나요?

### 예상 이해 수준
Point-to-point는 하나의 메시지를 하나의 consumer만 소비해 작업 분배에 적합하고, Pub/Sub는 하나의 메시지를 여러 subscriber가 독립적으로 소비해 이벤트 팬아웃에 적합함을 설명할 수 있어야 합니다. 주문 처리 워커는 point-to-point, 사용자 가입 이벤트에 반응하는 여러 서비스는 pub/sub가 적합함을 구체적인 예시로 설명할 수 있어야 합니다.

## Level 3: 심화

### 핵심 질문
- Message ordering 보장, exactly-once delivery, dead letter queue가 각각 무엇이고 왜 어려운 문제인지 설명해주세요.

### 꼬리 질문
- At-least-once delivery 환경에서 중복 메시지 처리를 어떻게 안전하게 처리할 수 있나요? (idempotency)
- Dead letter queue에 쌓인 메시지를 어떻게 운영상으로 처리해야 하나요?

### 예상 이해 수준
분산 환경에서 메시지 순서 보장은 파티션 내에서만 가능하고(Kafka의 partition key 활용), 전체 순서 보장은 처리량과 트레이드오프가 있음을 설명할 수 있어야 합니다. Exactly-once는 네트워크 장애와 재시도 때문에 구현이 어렵고 at-least-once + idempotent consumer 패턴으로 실용적으로 해결함을 설명할 수 있어야 합니다. Dead letter queue는 처리 불가한 메시지를 격리해 시스템이 막히지 않도록 하는 안전망임을 이해해야 합니다.

## Level 4: 시스템 설계

### 핵심 질문
- 대규모 이커머스 주문 처리 시스템을 event-driven architecture로 설계해주세요. 주문 생성부터 재고 차감, 결제, 배송 시작까지의 흐름을 message queue를 활용해 설계하세요.

### 꼬리 질문
- 결제 서비스가 일시적으로 다운된 경우 전체 주문 흐름에 어떤 영향이 있고, 어떻게 복구하나요?
- 각 단계가 모두 성공하거나 모두 실패해야 하는 분산 트랜잭션을 message queue 환경에서 어떻게 구현하나요? (Saga pattern)

### 예상 이해 수준
주문 생성 이벤트를 시작으로 재고 서비스, 결제 서비스, 배송 서비스가 각각 독립적으로 이벤트를 구독하고 처리하는 choreography 또는 orchestration 기반의 event-driven 흐름을 설계할 수 있어야 합니다. Saga pattern을 통해 분산 트랜잭션을 처리하고, 각 단계 실패 시 compensating transaction으로 롤백하는 메커니즘을 설명할 수 있어야 합니다.

## 현실 연결
- Apache Kafka를 활용한 실시간 로그 수집, 사용자 행동 분석, 마이크로서비스 간 이벤트 스트리밍
- RabbitMQ를 사용한 이메일 발송, 이미지 처리, 알림 발송 등 백그라운드 작업 처리
- AWS SQS와 SNS를 조합한 fan-out 패턴 및 Lambda와의 연동을 통한 서버리스 이벤트 처리
