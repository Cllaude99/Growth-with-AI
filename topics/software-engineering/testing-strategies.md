# Testing Strategies

## 메타데이터
- **카테고리**: 소프트웨어 공학
- **난이도**: 중급
- **우선순위**: 중간
- **선수과목**: 없음
- **예상 시간**: 30-45분

## Level 1: 기본 이해

### 핵심 질문
- Unit test, Integration test, E2E(End-to-End) test의 차이는 무엇인가요? 각각 어떤 것을 검증하고, 실행 속도와 비용 측면에서 어떤 차이가 있나요?

### 꼬리 질문
- Unit test에서 "unit"의 범위는 어디까지인가요? 메서드 하나인가요, 클래스 하나인가요?
- Integration test가 Unit test보다 더 신뢰할 수 있는 테스트라고 볼 수 있나요? 그렇다면 왜 Unit test가 필요한가요?

### 예상 이해 수준
각 테스트 레벨의 목적과 검증 범위를 명확히 구분할 수 있어야 합니다. Unit test는 외부 의존성을 격리하고 하나의 단위를 빠르게 검증하며, Integration test는 컴포넌트 간 상호작용을 검증하고, E2E test는 실제 사용자 시나리오를 전체 스택에서 검증함을 이해해야 합니다. 테스트 레벨이 높을수록 실행 속도가 느리고 유지보수 비용이 높아지는 트레이드오프도 알아야 합니다.

## Level 2: 트레이드오프

### 핵심 질문
- Testing pyramid(테스팅 피라미드)와 Testing trophy(테스팅 트로피)는 어떤 차이가 있나요? 각 모델이 권장하는 테스트 비율과 그 근거를 설명하고, 프로젝트 상황에 따라 어떤 모델을 선택하겠나요?

### 꼬리 질문
- Testing trophy에서 Integration test를 강조하는 이유는 무엇인가요? Unit test 중심의 테스트 스위트의 한계는 무엇인가요?
- E2E test가 불안정(flaky)한 경우가 많은 이유는 무엇이고, 어떻게 안정성을 높일 수 있나요?

### 예상 이해 수준
Testing pyramid는 Unit test를 기반으로 빠르고 많은 테스트를 구성하는 전통적 관점이고, Testing trophy는 Integration test의 신뢰성을 중시하는 Kent C. Dodds의 접근임을 이해해야 합니다. "테스트는 사용자가 소프트웨어를 사용하는 방식과 유사할수록 신뢰도가 높다"는 원칙을 이해하고, 팀 규모, 서비스 성격, CI 실행 시간 등 실제 맥락에 따라 전략을 선택할 수 있어야 합니다.

## Level 3: 심화

### 핵심 질문
- Mocking의 장단점은 무엇인가요? Test double의 종류인 Stub, Spy, Mock, Fake의 차이를 설명하고, 각각 어떤 상황에서 사용해야 하나요?

### 꼬리 질문
- "Mock을 많이 사용하면 테스트가 구현에 종속된다"는 말의 의미는 무엇인가요? 이를 방지하기 위한 전략이 있나요?
- Fake 객체가 Stub보다 더 현실적인 테스트를 가능하게 하는 이유는 무엇인가요? Fake를 사용할 때의 위험성은 무엇인가요?

### 예상 이해 수준
각 Test double의 차이를 명확히 설명할 수 있어야 합니다. Stub은 미리 정의된 응답을 반환하고, Spy는 실제 동작을 유지하면서 호출을 기록하며, Mock은 기대 동작을 사전에 정의하고 검증하고, Fake는 실제 동작하는 단순한 대체 구현체임을 이해해야 합니다. 과도한 Mocking이 테스트가 리팩토링에 취약해지는 "brittleness" 문제를 만드는 이유를 설명할 수 있어야 합니다.

## Level 4: 시스템 설계

### 핵심 질문
- 테스트 코드가 전혀 없는 레거시 시스템에 테스트를 도입해야 합니다. 어디서부터 시작하겠나요? TDD를 새로 도입하려는데 팀의 저항이 있다면 어떻게 접근하겠나요?

### 꼬리 질문
- 레거시 코드에 테스트를 추가하기 어려운 이유는 무엇인가요? 테스트 작성을 어렵게 만드는 코드의 특성은 무엇인가요?
- TDD의 Red-Green-Refactor 사이클이 단순히 "테스트 먼저 작성"이 아닌 설계 도구로서 어떤 가치가 있나요?

### 예상 이해 수준
레거시 코드에서는 가장 변경이 잦거나 버그가 많은 영역부터 "characterization test"로 현재 동작을 문서화하는 접근부터 시작함을 알아야 합니다. TDD 도입 시 팀 전체가 아닌 작은 단위부터 시작하고, 테스트 가능한 설계(dependency injection, 작은 함수 등)가 결과적으로 더 좋은 아키텍처를 만든다는 것을 설명할 수 있어야 합니다.

## 현실 연결
- JUnit(Java)과 Jest(JavaScript)는 Unit test 프레임워크의 대표로, assertion 라이브러리와 Mocking 기능을 제공하며 CI 파이프라인에 통합되어 PR 시 자동 실행됨
- Cypress와 Playwright는 E2E test 도구로, 실제 브라우저에서 사용자 시나리오를 자동화하며 시각적 회귀 테스트와 결합해 UI 변경 감지에 활용됨
- TestContainers는 Docker를 활용해 테스트 시 실제 DB나 메시지 큐를 격리된 환경에서 실행하여, In-memory DB보다 현실적인 Integration test를 가능하게 하고 CI/CD 파이프라인에서도 동일한 환경을 보장함
