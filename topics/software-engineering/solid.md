# SOLID

## 메타데이터
- **카테고리**: 소프트웨어 공학
- **난이도**: 중급
- **우선순위**: 높음
- **선수과목**: OOP Principles
- **예상 시간**: 30-45분

## Level 1: 기본 이해

### 핵심 질문
- SOLID의 각 원칙인 SRP, OCP, LSP, ISP, DIP가 무엇인지 설명하고, 각 원칙이 해결하려는 핵심 문제가 무엇인지 말해주세요.

### 꼬리 질문
- SRP에서 "하나의 책임"이란 무엇인가요? 책임의 경계를 어떻게 판단하나요?
- DIP에서 "추상화에 의존해야 한다"는 말의 의미를 구체적인 예시로 설명해주세요.

### 예상 이해 수준
각 원칙의 약자와 정의를 외우는 수준을 넘어서, 원칙이 존재하는 이유와 각 원칙이 어떤 설계 문제를 예방하는지 설명할 수 있어야 합니다. 예를 들어 OCP는 기존 코드 수정 없이 새로운 기능을 추가할 수 있도록 하여 버그 유입을 줄이고, LSP는 서브타입이 기반 타입을 완전히 대체할 수 있어야 한다는 것을 이해해야 합니다.

## Level 2: 트레이드오프

### 핵심 질문
- SOLID 원칙 중 하나 이상을 위반하면 실제로 어떤 문제가 발생하나요? 각 원칙 위반의 증상과 그로 인한 결과를 구체적으로 설명해주세요.

### 꼬리 질문
- SRP를 지나치게 적용해서 클래스를 지나치게 잘게 쪼갠다면 어떤 문제가 생기나요?
- LSP를 위반하는 코드를 작성했을 때, 테스트에서 발견하기 어려운 이유는 무엇인가요?

### 예상 이해 수준
단순히 "유지보수가 어렵다"는 수준이 아니라, 원칙 위반이 가져오는 구체적인 증상을 설명할 수 있어야 합니다. 예를 들어 OCP 위반 시 기능 추가마다 기존 코드를 수정해야 해서 side effect가 발생하고 회귀 테스트 비용이 증가하는 것, ISP 위반 시 불필요한 메서드를 구현하거나 빈 메서드 또는 예외를 던지는 코드가 생기는 것을 설명해야 합니다.

## Level 3: 심화

### 핵심 질문
- SOLID 원칙들이 서로 충돌하는 상황이 있나요? 예를 들어 SRP를 지키면 클래스가 많아지는데 이것이 다른 원칙과 충돌할 수 있는지, 우선순위를 어떻게 판단하나요?

### 꼬리 질문
- "모든 추상화에는 비용이 있다"는 말이 있는데, DIP를 위해 인터페이스를 남발하면 어떤 문제가 생기나요?
- LSP와 OCP는 어떤 관계가 있나요? 두 원칙이 함께 지켜져야 하는 이유를 설명해주세요.

### 예상 이해 수준
원칙들이 이상적인 목표이지 절대적인 규칙이 아님을 이해해야 합니다. 문맥(context)과 팀 규모, 프로젝트 수명 주기에 따라 원칙 적용의 강도가 달라질 수 있음을 알고, 단순한 CRUD 기능에 OCP를 위해 과도한 추상화를 도입하는 것이 오히려 코드 복잡성을 높이는 YAGNI 위반임을 판단할 수 있어야 합니다.

## Level 4: 시스템 설계

### 핵심 질문
- 아래 레거시 코드 패턴을 SOLID 원칙에 맞게 리팩토링해야 합니다. 여러 가지 비즈니스 로직(이메일 발송, 데이터 저장, 알림 전송)이 하나의 클래스에 섞여 있고, 새로운 알림 채널을 추가할 때마다 if-else가 늘어나는 구조입니다. 어떤 순서로 어떻게 리팩토링하겠나요?

### 꼬리 질문
- 리팩토링 중 기존 기능이 깨지지 않는 것을 어떻게 보장하나요?
- 리팩토링 후 코드의 테스트 용이성이 어떻게 달라지나요? 그 이유는 무엇인가요?

### 예상 이해 수준
SRP에 따라 각 책임을 별도 클래스로 분리하고, OCP에 따라 알림 채널을 인터페이스로 추상화하여 새 채널 추가 시 기존 코드 수정 없이 구현체만 추가하도록 리팩토링하는 과정을 설명할 수 있어야 합니다. DIP를 통해 고수준 모듈이 저수준 구현에 의존하지 않도록 구조를 잡고, 단계적 리팩토링을 통해 안전하게 변경하는 전략을 제시해야 합니다.

## 현실 연결
- Spring Framework는 DIP를 핵심으로, 인터페이스 기반의 DI 컨테이너를 통해 Service가 Repository 구현체가 아닌 인터페이스에 의존하도록 설계되어 있어 테스트 시 Mock 객체로 쉽게 교체 가능함
- Clean Architecture에서 SOLID는 레이어 경계 설계의 기반으로, 특히 DIP는 외부 의존성(DB, 외부 API)이 내부 비즈니스 로직에 영향을 주지 않도록 의존성 방향을 역전시킴
- Plugin architecture(VSCode 확장, IntelliJ 플러그인)는 OCP의 대표적 실사례로, 핵심 애플리케이션 코드를 수정하지 않고 외부 플러그인을 추가해 기능을 확장할 수 있도록 설계됨
