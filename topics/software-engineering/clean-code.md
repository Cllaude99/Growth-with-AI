# Clean Code

## 메타데이터
- **카테고리**: 소프트웨어 공학
- **난이도**: 기초
- **우선순위**: 중간
- **선수과목**: 없음
- **예상 시간**: 30-45분

## Level 1: 기본 이해

### 핵심 질문
- Clean code란 무엇인가요? 단순히 "잘 작동하는 코드"와 "clean한 코드"는 어떻게 다르고, clean code가 왜 중요한지 설명해주세요.

### 꼬리 질문
- "나중에 읽을 사람을 위해 코드를 작성하라"는 말이 있는데, 그 "나중에 읽을 사람"이 미래의 자신일 수도 있습니다. 이것이 실무에서 어떤 의미를 가지나요?
- Clean code와 "작동하는 코드"가 상충될 수 있는 상황은 어떤 경우인가요? 마감 기한이 촉박할 때 어떤 선택을 하겠나요?

### 예상 이해 수준
Clean code는 읽기 쉽고, 이해하기 쉽고, 변경하기 쉬운 코드임을 이해해야 합니다. 소프트웨어의 생명 주기에서 코드를 읽는 시간이 쓰는 시간보다 훨씬 많다는 사실, 그리고 코드는 한 번 작성되지만 수십 번 읽힌다는 것을 인식하고, 가독성이 단순한 스타일 문제가 아닌 팀 생산성과 직결된다는 것을 설명할 수 있어야 합니다.

## Level 2: 트레이드오프

### 핵심 질문
- 가독성(readability), 성능(performance), 간결함(conciseness)은 때로 서로 충돌합니다. 각각의 트레이드오프를 설명하고, 실제 코드를 작성할 때 어떤 우선순위로 접근하나요?

### 꼬리 질문
- "조기 최적화는 모든 악의 근원이다(Premature optimization is the root of all evil)"는 말의 의미는 무엇이고, 실무에서 이 원칙을 어떻게 적용하나요?
- 코드가 매우 간결하지만 이해하기 어려운 경우(예: 복잡한 one-liner)와 길지만 명확한 코드 중 어떤 것이 더 clean한 코드인가요?

### 예상 이해 수준
성능 최적화는 프로파일링을 통해 실제 병목이 확인된 후에 적용해야 하며, 명확성을 해치는 최적화는 주석과 함께 이유를 문서화해야 함을 이해해야 합니다. 간결함이 가독성을 해칠 때는 명확함을 선택해야 한다는 것을 알고, 컨텍스트(팀의 수준, 코드베이스의 성격)에 따라 판단 기준이 달라질 수 있음을 설명할 수 있어야 합니다.

## Level 3: 심화

### 핵심 질문
- 좋은 네이밍의 원칙, 함수의 적절한 크기, 주석을 작성해야 할 때와 피해야 할 때, 그리고 에러 처리의 원칙에 대해 설명해주세요. 이 중 실무에서 가장 자주 위반되는 것은 무엇이라고 생각하나요?

### 꼬리 질문
- "주석은 코드가 설명하지 못하는 것만 적어야 한다"는 말이 있는데, 어떤 종류의 주석이 가치 있고 어떤 주석이 오히려 해가 되나요?
- 함수가 한 가지 일만 해야 한다(SRP)는 원칙과 함수의 추상화 수준이 일관돼야 한다는 원칙의 관계를 설명해주세요.

### 예상 이해 수준
이름은 의도를 드러내야 하며 검색 가능하고 발음 가능해야 함을 이해해야 합니다. 함수는 한 가지 일을 하고 20줄 이내를 권장하지만 이것이 절대 기준이 아님을 알아야 합니다. 주석은 "왜(why)"를 설명하고 "무엇(what)"은 코드가 설명해야 하며, 에러는 무시하지 말고 의미 있는 메시지와 함께 처리해야 함을 구체적인 좋은 예시와 나쁜 예시와 함께 설명할 수 있어야 합니다.

## Level 4: 시스템 설계

### 핵심 질문
- 팀의 코드 리뷰 문화를 개선하고, 기술 부채를 체계적으로 관리하는 리팩토링 전략을 수립해야 합니다. 코드 리뷰에서 어떤 기준으로 피드백을 주겠나요? 기술 부채를 어떻게 식별하고 우선순위를 정하겠나요?

### 꼬리 질문
- 코드 리뷰에서 스타일(style)과 설계(design) 피드백을 어떻게 구분하고, 어떤 것에 더 집중해야 하나요?
- "Boy Scout Rule(보이스카우트 규칙: 왔을 때보다 더 깨끗하게 떠나라)"을 팀에 어떻게 적용하겠나요? 이것이 지속적인 리팩토링 문화와 어떻게 연결되나요?

### 예상 이해 수준
코드 리뷰는 버그 발견뿐 아니라 지식 공유와 팀 코드 품질 기준 수립의 역할을 함을 이해해야 합니다. 기술 부채는 의도적인 것(빠른 출시를 위한 임시 방편)과 비의도적인 것(무지로 인한 나쁜 설계)을 구분하고, 부채의 이자(유지보수 비용)를 측정해 우선순위를 정하는 접근을 설명할 수 있어야 합니다.

## 현실 연결
- Code review 문화는 Google, Airbnb 등의 기업에서 모든 코드 변경이 최소 한 명의 다른 개발자 리뷰를 통과해야 머지되도록 하는 의무적 프로세스로 운영되며, PR 크기를 작게 유지하는 것이 효과적인 리뷰의 핵심임
- ESLint, Prettier, Black(Python), ktlint(Kotlin) 같은 Linter/Formatter는 스타일 논쟁을 자동화로 해결하고, 코드 리뷰에서 "탭 vs 스페이스" 같은 소모적 논쟁을 제거하여 팀이 설계 품질에 집중할 수 있게 함
- 기술 부채 관리는 Sonarqube 같은 정적 분석 도구로 코드 smell을 자동 탐지하고, "부채 상환 스프린트"를 정기적으로 배정하거나 새 기능 개발과 리팩토링을 일정 비율로 섞어 진행하는 방식으로 관리됨
