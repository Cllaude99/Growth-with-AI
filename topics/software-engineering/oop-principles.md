# OOP Principles

## 메타데이터
- **카테고리**: 소프트웨어 공학
- **난이도**: 기초
- **우선순위**: 높음
- **선수과목**: 없음
- **예상 시간**: 30-45분

## Level 1: 기본 이해

### 핵심 질문
- OOP의 4대 특성인 캡슐화, 상속, 다형성, 추상화가 각각 무엇인지 설명하고, 왜 이 개념들이 소프트웨어 설계에 중요한지 말해주세요.

### 꼬리 질문
- 캡슐화를 통해 얻는 이점이 단순히 데이터를 숨기는 것 외에 또 무엇이 있나요?
- 추상화와 캡슐화는 어떻게 다른가요? 두 개념이 혼동되는 이유는 무엇이라고 생각하나요?

### 예상 이해 수준
각 특성의 정의를 단순 암기 수준을 넘어, 실제 코드 예시를 들어 설명할 수 있어야 합니다. 예를 들어 캡슐화는 접근 제어자(private/public)를 통한 정보 은닉과 객체의 상태 보호임을 구체적으로 설명하고, 상속은 코드 재사용과 계층 구조 표현, 다형성은 동일한 인터페이스로 다른 동작을 가능하게 함을 이해해야 합니다.

## Level 2: 트레이드오프

### 핵심 질문
- Composition(구성)과 Inheritance(상속) 중 어떤 상황에서 무엇을 선택해야 하나요? 각각의 장단점을 비교해서 설명해주세요.

### 꼬리 질문
- "Favor composition over inheritance"라는 원칙이 있는데, 이 말이 상속을 절대 쓰지 말라는 의미인가요? 상속이 적합한 경우는 언제인가요?
- 깊은 상속 계층(deep inheritance hierarchy)이 야기하는 구체적인 문제는 무엇인가요?

### 예상 이해 수준
단순히 "is-a 관계면 상속, has-a 관계면 composition"을 넘어서, 상속이 강한 결합(tight coupling)을 만들고 변경에 취약해지는 이유를 설명할 수 있어야 합니다. Composition이 유연성을 높이는 메커니즘을 이해하고, 실제 코드 구조에서 두 방식의 트레이드오프를 판단할 수 있어야 합니다.

## Level 3: 심화

### 핵심 질문
- 다형성의 동적 바인딩(dynamic binding)이 어떻게 동작하는지 설명하고, abstract class와 interface의 차이 및 각각을 선택하는 기준을 설명해주세요.

### 꼬리 질문
- Java에서 abstract class는 단일 상속만 가능하고 interface는 다중 구현이 가능한데, 이 제약이 설계에 어떤 영향을 미치나요?
- Duck typing(Python 등)이 적용되는 언어에서는 interface 없이도 다형성을 어떻게 구현하나요? 이 방식의 장단점은 무엇인가요?

### 예상 이해 수준
동적 바인딩이 vtable(가상 함수 테이블)을 통해 런타임에 메서드를 결정하는 메커니즘을 이해해야 합니다. Abstract class는 부분적인 구현을 제공하며 공통 상태(필드)를 가질 수 있는 반면, interface는 계약(contract)만을 정의함을 알고, 언어와 요구사항에 따라 적절히 선택할 수 있어야 합니다.

## Level 4: 시스템 설계

### 핵심 질문
- 전자상거래 플랫폼에서 다양한 상품 유형(디지털 상품, 실물 상품, 구독 상품)을 처리하는 클래스 구조를 OOP 원칙을 적용해서 설계해보세요. 어떤 선택을 했고 왜 그렇게 했나요?

### 꼬리 질문
- 나중에 새로운 상품 유형이 추가될 것을 고려하면, 지금 설계에서 어떤 부분이 변경에 가장 취약한가요? 어떻게 개선하겠나요?
- 결제 처리 로직은 어떤 클래스에 두어야 할까요? 상품 클래스 안에 두는 것이 맞나요, 아니면 별도로 분리해야 하나요?

### 예상 이해 수준
추상화를 통해 Product 기반 클래스나 인터페이스를 정의하고, 상품 유형별 특성을 상속이나 composition으로 표현하는 설계를 제안할 수 있어야 합니다. 단순히 클래스 다이어그램을 그리는 것이 아니라, 각 설계 결정의 근거와 트레이드오프를 설명하고 변경에 열려 있는 구조를 의식적으로 설계할 수 있어야 합니다.

## 현실 연결
- Java Spring의 DI(Dependency Injection) 컨테이너는 인터페이스 기반 다형성을 활용해, 구체 클래스를 런타임에 주입함으로써 결합도를 낮추고 테스트 용이성을 높임
- Python의 duck typing은 명시적 인터페이스 없이 동일한 메서드 시그니처만 가지면 다형성을 구현할 수 있어 유연하지만, 타입 안정성이 낮아 런타임 오류 가능성이 있음
- React의 component hierarchy는 composition 패턴을 UI 영역에 적용한 사례로, 작은 컴포넌트를 조합해 복잡한 UI를 구성하며 props를 통한 데이터 흐름이 캡슐화 역할을 함
