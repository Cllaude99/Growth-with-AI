# Design Patterns

## 메타데이터
- **카테고리**: 소프트웨어 공학
- **난이도**: 중급
- **우선순위**: 중간
- **선수과목**: OOP Principles, SOLID
- **예상 시간**: 30-45분

## Level 1: 기본 이해

### 핵심 질문
- Design pattern이란 무엇이고, GoF(Gang of Four)가 분류한 생성(Creational), 구조(Structural), 행위(Behavioral) 패턴 카테고리 각각의 목적과 대표 패턴을 설명해주세요.

### 꼬리 질문
- Design pattern을 사용하면 반드시 코드가 좋아지나요? 패턴을 배워야 하는 진짜 이유는 무엇인가요?
- Singleton 패턴을 설명하고, 그것이 생성 패턴으로 분류되는 이유를 설명해주세요.

### 예상 이해 수준
Design pattern이 특정 문제에 대한 재사용 가능한 설계 솔루션이며, 코드 조각이 아닌 "구조적 아이디어"임을 이해해야 합니다. 세 카테고리의 목적(생성: 객체 생성 방식, 구조: 클래스/객체 조합, 행위: 객체 간 책임/통신)을 구분하고, 개발자 간 공통 언어로서 패턴 이름이 커뮤니케이션 비용을 줄이는 역할을 한다는 것을 알아야 합니다.

## Level 2: 트레이드오프

### 핵심 질문
- Strategy 패턴과 State 패턴은 구조가 비슷한데 어떤 차이가 있나요? Factory 패턴과 Builder 패턴은 각각 어떤 상황에서 선택해야 하나요?

### 꼬리 질문
- Strategy 패턴에서 알고리즘 교체가 런타임에 일어난다면, State 패턴과의 차이는 상태 전이 로직을 어디에 두느냐 외에 또 무엇인가요?
- Builder 패턴을 사용하는 객체와 단순 생성자(constructor) 또는 팩토리 메서드를 사용하는 객체의 차이를 코드 수준에서 설명해주세요.

### 예상 이해 수준
Strategy와 State는 구조가 동일하지만, Strategy는 클라이언트가 명시적으로 알고리즘을 선택하고 상태 전이가 없는 반면, State는 상태 자체가 전이 로직을 담당하고 객체의 내부 상태에 따라 행동이 바뀜을 설명할 수 있어야 합니다. Factory는 어떤 타입을 생성할지 결정에, Builder는 복잡한 객체의 단계적 생성에 초점이 있음을 실제 예시와 함께 설명해야 합니다.

## Level 3: 심화

### 핵심 질문
- Anti-pattern이란 무엇이고, Design pattern을 남용하거나 잘못 적용했을 때 어떤 문제가 생기나요? 실제로 경험했거나 코드에서 본 적 있는 anti-pattern 사례를 설명해주세요.

### 꼬리 질문
- Singleton 패턴은 왜 anti-pattern으로 여겨지기도 하나요? Singleton이 테스트를 어렵게 만드는 구체적인 이유는 무엇인가요?
- "Over-engineering"과 "Design pattern 남용"의 관계를 설명하고, 패턴 적용이 적절한지 판단하는 기준이 있나요?

### 예상 이해 수준
"패턴을 위한 패턴"이 오히려 코드 복잡성을 높이고 이해를 어렵게 만든다는 것을 이해해야 합니다. Singleton이 전역 상태를 만들어 테스트 격리를 어렵게 하고, 숨겨진 의존성을 만드는 문제를 설명할 수 있어야 합니다. "패턴은 문제를 먼저 인식하고 나서 솔루션으로 적용해야 한다"는 원칙을 이해하고, 단순한 문제에 복잡한 패턴을 도입하는 것이 YAGNI 위반임을 판단할 수 있어야 합니다.

## Level 4: 시스템 설계

### 핵심 질문
- 알림 시스템을 설계한다고 가정합시다. 이메일, SMS, 푸시 알림, 슬랙 등 다양한 채널과 긴급 알림, 마케팅 알림 등 다양한 알림 유형을 처리해야 합니다. 어떤 Design pattern을 조합해서 설계하겠나요? 각 패턴 선택의 이유를 설명해주세요.

### 꼬리 질문
- 새로운 알림 채널을 추가하거나 알림 유형을 추가할 때 기존 코드를 얼마나 수정해야 하나요? 이 설계가 OCP를 만족하나요?
- 알림 발송 실패 시 재시도 로직을 추가해야 한다면, 어떤 패턴을 사용하겠나요?

### 예상 이해 수준
Strategy 패턴으로 알림 채널을 추상화하고, Factory 또는 Abstract Factory로 채널 인스턴스를 생성하며, Observer 패턴으로 이벤트 발생 시 알림을 트리거하는 구조를 설계할 수 있어야 합니다. 패턴을 조합할 때 각 패턴이 어떤 책임을 담당하는지 명확히 설명하고, 확장성과 유지보수성을 고려한 설계를 제시해야 합니다.

## 현실 연결
- Observer 패턴은 DOM 이벤트 시스템, Node.js EventEmitter, RxJS Observable 등 이벤트 기반 시스템 전반에 적용되어 발행자(publisher)와 구독자(subscriber)를 분리함
- Strategy 패턴은 결제 시스템에서 신용카드, 간편결제, 가상계좌 등 다양한 결제 수단을 동일한 인터페이스로 처리하는 데 사용되며, 런타임에 결제 전략을 교체할 수 있게 함
- Singleton 패턴은 DB 커넥션 풀 관리에 활용되어 애플리케이션 전체에서 하나의 커넥션 풀 인스턴스를 공유하지만, 현대에는 DI 컨테이너가 이를 대체하며 테스트 용이성을 높임
