# Syncspot

## 메타데이터
- **카테고리**: 이력서 면접
- **난이도**: 중급
- **우선순위**: 최고
- **선수과목**: 없음
- **예상 시간**: 30-45분
- **프로젝트 개요**: 위치 기반 모임 서비스 (React, TypeScript, TanStack Query, TailwindCSS)

## Level 1: 프로젝트 개요 & 기술 선택

### 핵심 질문
- Syncspot 프로젝트에서 본인의 역할과 기술 스택 선택 이유를 설명해주세요. 특히 TanStack Query를 선택한 이유가 무엇이었나요?

### 꼬리 질문
- TanStack Query 대신 SWR이나 Redux Toolkit Query를 사용할 수도 있었을 텐데, 왜 TanStack Query였나요?
- TailwindCSS를 선택한 이유는 무엇이었나요? styled-components나 CSS Modules과 비교했을 때 어떤 점이 좋았나요?

### 예상 이해 수준
프로젝트의 핵심 기능과 자신의 기여를 명확히 설명할 수 있어야 한다. 기술 선택 시 고려한 기준(러닝커브, 팀 컨벤션, 프로젝트 특성 등)을 구체적으로 제시할 수 있어야 한다. 단순히 "인기 있어서"가 아닌 프로젝트 맥락에 맞는 선택 근거를 설명할 수 있어야 한다.

## Level 2: 문제 해결 과정

### 핵심 질문
- Syncspot에서 가장 어려웠던 기술적 문제는 무엇이었고, 어떤 과정을 거쳐 해결했나요? 그 해결 방법을 선택한 이유는 무엇이었나요?

### 꼬리 질문
- 디바운스를 적용해서 API 호출을 최적화했다고 했는데, 디바운스 대신 쓰로틀을 사용하지 않은 이유는 무엇인가요? 각각 어떤 상황에 더 적합한가요?
- Error Boundary를 도입한 계기는 무엇이었나요? Error Boundary 없이 try-catch만으로 에러 처리를 하면 어떤 문제가 생기나요?
- SRP 기반으로 700줄을 400줄로 리팩토링했다고 했는데, 어떤 기준으로 컴포넌트를 분리했나요?
- Sentry를 도입하게 된 계기는 무엇이고, 도입 전후로 에러 대응 방식이 어떻게 달라졌나요?
- 지도 bounds 최적화는 구체적으로 어떤 문제를 해결한 건가요?

### 예상 이해 수준
문제를 인식한 경위(사용자 피드백, 성능 측정, 코드 리뷰 등)를 설명할 수 있어야 한다. 여러 해결 방안 중 선택한 방법의 근거를 대안과 비교하여 제시할 수 있어야 한다. 디바운스 vs 쓰로틀의 동작 차이와 적합한 사용 케이스를 구분할 수 있어야 한다. Error Boundary의 React 컴포넌트 트리 레벨 에러 핸들링 특성을 이해하고 있어야 한다.

## Level 3: 확장 & 엣지케이스

### 핵심 질문
- Syncspot의 사용자가 10배로 늘어난다면, 현재 아키텍처에서 가장 먼저 병목이 될 부분은 어디이고, 어떻게 대응하겠나요?

### 꼬리 질문
- 지도에 표시할 마커가 수천 개가 된다면 현재 설계로 감당 가능한가요? 어떤 최적화가 필요할까요?
- TanStack Query의 캐싱 전략을 어떻게 설정했나요? staleTime과 cacheTime을 어떤 기준으로 결정했나요?
- 드래그 바텀시트 구현에서 터치 이벤트와 스크롤 이벤트가 충돌하는 경우는 없었나요? 있었다면 어떻게 해결했나요?
- CI/CD 파이프라인은 어떻게 구성했나요? 배포 과정에서 롤백이 필요한 상황이 발생하면 어떻게 대응하나요?

### 예상 이해 수준
현재 아키텍처의 한계점을 객관적으로 인식하고, 스케일링 시 발생할 수 있는 문제를 구체적으로 예측할 수 있어야 한다. 지도 기반 서비스의 성능 병목(렌더링, 네트워크, 데이터 크기)을 이해하고 클러스터링, 가상화 등의 대응 방안을 제시할 수 있어야 한다. 캐싱 전략의 트레이드오프를 이해하고 있어야 한다.

## Level 4: 설계 판단 & 아키텍처

### 핵심 질문
- Syncspot을 처음부터 다시 설계한다면, 기술 스택이나 아키텍처에서 어떤 부분을 다르게 가져가겠나요? 그 이유는 무엇인가요?

### 꼬리 질문
- 공통 컴포넌트를 설계할 때 어떤 원칙을 적용했나요? 재사용성과 유연성 사이의 균형을 어떻게 잡았나요?
- 프로젝트에서 상태 관리를 어떻게 했나요? 전역 상태와 서버 상태를 분리한 기준은 무엇이었나요?
- 팀에서 기술적 의사결정을 할 때 어떤 과정을 거쳤나요? 의견 충돌이 있었다면 어떻게 해결했나요?

### 예상 이해 수준
프로젝트 경험을 바탕으로 더 나은 설계 방향을 논리적으로 제시할 수 있어야 한다. 기술 부채와 초기 설계 결정의 영향을 인식하고, 트레이드오프를 고려한 대안을 제시할 수 있어야 한다. 팀 협업과 기술적 의사결정 과정을 구체적으로 설명할 수 있어야 한다.

## 현실 연결
- 위치 기반 서비스는 카카오맵, 네이버 지도, Google Maps 등 대규모 서비스에서도 핵심 기술이며, bounds 최적화와 마커 클러스터링은 실무에서 필수적인 최적화 기법이다.
- 디바운스/쓰로틀은 검색 자동완성, 스크롤 이벤트 처리, 윈도우 리사이즈 등 프론트엔드 전반에서 광범위하게 사용되는 최적화 패턴이다.
- Error Boundary와 Sentry를 결합한 에러 모니터링은 프로덕션 환경에서 서비스 안정성을 보장하는 표준적인 접근 방식이다.
