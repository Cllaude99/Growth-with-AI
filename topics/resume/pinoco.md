# Pinoco

## 메타데이터
- **카테고리**: 이력서 면접
- **난이도**: 중급
- **우선순위**: 최고
- **선수과목**: 없음
- **예상 시간**: 30-45분
- **프로젝트 개요**: 실시간 라이어 게임 서비스 (React, TypeScript, Socket.IO, WebRTC)

## Level 1: 프로젝트 개요 & 기술 선택

### 핵심 질문
- Pinoco 프로젝트에서 실시간 통신이 핵심이었을 텐데, Socket.IO를 선택한 이유는 무엇이었나요? 다른 대안은 고려하지 않았나요?

### 꼬리 질문
- WebSocket과 Socket.IO의 차이는 무엇인가요? 네이티브 WebSocket 대신 Socket.IO를 쓴 이유가 있나요?
- WebRTC는 어떤 기능에 사용했나요? Socket.IO만으로는 해결할 수 없었던 부분이 있었나요?
- MSW를 사용해서 목업 기반 UI 개발을 했다고 했는데, 이 방식의 장점은 무엇이었나요?

### 예상 이해 수준
Socket.IO가 WebSocket 위에 구축된 라이브러리로 자동 재연결, fallback, room/namespace 등의 추가 기능을 제공한다는 점을 이해하고 있어야 한다. WebRTC의 P2P 특성과 Socket.IO의 서버 중계 방식의 차이를 설명할 수 있어야 한다. MSW를 통한 API 모킹이 프론트엔드-백엔드 병렬 개발에 어떻게 기여하는지 설명할 수 있어야 한다.

## Level 2: 문제 해결 과정

### 핵심 질문
- Pinoco에서 성능 최적화를 여러 가지 했는데, 각 최적화가 필요했던 구체적인 상황과 측정 결과를 설명해줄 수 있나요?

### 꼬리 질문
- React.lazy와 Suspense를 사용한 코드 스플리팅은 어떤 문제를 해결하기 위한 것이었나요? 번들 사이즈가 얼마나 줄었나요?
- React.memo를 사용해서 리렌더링을 최적화했다고 했는데, 어떤 컴포넌트에 적용했고, 리렌더링이 발생하는 원인은 무엇이었나요?
- requestAnimationFrame으로 타이머를 구현한 이유는 무엇인가요? setInterval 대신 rAF를 선택한 이유가 있나요?
- useRef로 채팅 스크롤을 관리한 방식은 어떤 건가요? useState 대신 useRef를 쓴 이유는 무엇인가요?

### 예상 이해 수준
코드 스플리팅의 원리와 초기 로딩 성능에 미치는 영향을 설명할 수 있어야 한다. React.memo의 동작 방식과 적용 기준(props 비교, 렌더링 비용)을 이해하고 있어야 한다. rAF와 setInterval의 차이(브라우저 렌더링 사이클 동기화, 탭 비활성화 시 동작)를 설명할 수 있어야 한다. useRef가 리렌더링을 유발하지 않는다는 특성과 DOM 접근 용도를 이해하고 있어야 한다.

## Level 3: 확장 & 엣지케이스

### 핵심 질문
- 실시간 게임에서 네트워크 불안정이나 사용자 연결 끊김 같은 엣지케이스를 어떻게 처리했나요? 처리하지 못한 부분이 있다면 어떻게 개선하겠나요?

### 꼬리 질문
- 동시 접속자가 1000명이 된다면 Socket.IO 서버 아키텍처를 어떻게 확장하겠나요?
- 게임 중 한 명이 나가면 어떻게 처리했나요? 재접속은 가능했나요?
- 실시간 통신에서 메시지 순서가 보장되지 않는 상황이 발생할 수 있는데, 이런 경우를 고려했나요?
- WebRTC 연결이 실패하는 경우(방화벽, NAT 등)에 대한 fallback은 어떻게 처리했나요?

### 예상 이해 수준
Socket.IO의 재연결 메커니즘과 연결 상태 관리를 이해하고 있어야 한다. 실시간 서비스의 스케일링 전략(Redis adapter, 로드밸런서 sticky session 등)을 인식하고 있어야 한다. 게임 상태 관리에서 발생할 수 있는 동시성 문제와 이를 해결하기 위한 전략을 설명할 수 있어야 한다. WebRTC의 STUN/TURN 서버 역할을 이해하고 있어야 한다.

## Level 4: 설계 판단 & 아키텍처

### 핵심 질문
- Pinoco를 처음부터 다시 설계한다면, 실시간 통신 아키텍처와 상태 관리를 어떻게 다르게 가져가겠나요?

### 꼬리 질문
- 게임 로직을 클라이언트에서 처리했나요, 서버에서 처리했나요? 그 결정의 이유와 장단점은 무엇인가요?
- 실시간 게임의 상태를 서버와 클라이언트 간에 어떻게 동기화했나요? 상태 불일치가 발생하면 어떻게 해결했나요?
- Socket.IO 이벤트 구조를 어떻게 설계했나요? 이벤트가 많아지면 관리가 어려워질 수 있는데, 어떤 패턴을 적용했나요?
- 팀원들과 실시간 통신 프로토콜을 어떻게 정의하고 문서화했나요?

### 예상 이해 수준
클라이언트-서버 간 게임 상태 관리의 설계 결정과 그 트레이드오프를 논리적으로 설명할 수 있어야 한다. 실시간 통신 아키텍처의 다양한 패턴(이벤트 기반, pub/sub, 상태 동기화)을 비교할 수 있어야 한다. 프로젝트 경험을 바탕으로 개선 방향을 구체적으로 제시할 수 있어야 한다.

## 현실 연결
- Socket.IO는 Slack, Trello 등 실시간 협업 도구에서 널리 사용되며, 대규모 서비스에서는 Redis adapter를 통한 수평 확장이 표준적인 패턴이다.
- WebRTC는 Google Meet, Discord 등 실시간 음성/영상 서비스의 핵심 기술이며, STUN/TURN 서버를 통한 NAT traversal은 P2P 통신의 필수 인프라이다.
- 코드 스플리팅과 React.memo를 통한 최적화는 프론트엔드 성능 개선의 기본이며, Lighthouse나 React DevTools Profiler로 측정하고 개선하는 것이 실무 표준이다.
- MSW는 프론트엔드 테스팅과 개발 환경 구축에서 점점 더 많이 채택되고 있는 도구이다.
