# Normalization

## 메타데이터
- **카테고리**: 데이터베이스
- **난이도**: 중급
- **우선순위**: 중간
- **선수과목**: SQL Join
- **예상 시간**: 30-45분

## Level 1: 기본 이해

### 핵심 질문
- 정규화의 목적은 무엇이고, 1NF, 2NF, 3NF는 각각 어떤 조건을 요구하는가?

### 꼬리 질문
- 정규화가 되지 않은 테이블에서 발생하는 이상 현상(anomaly)에는 어떤 것들이 있는가?
- 1NF에서 "원자값"이란 무엇인가? 배열이나 JSON을 컬럼에 저장하면 1NF를 위반하는가?

### 예상 이해 수준
정규화는 데이터 중복을 제거하고 삽입/수정/삭제 이상 현상을 방지하기 위한 테이블 설계 원칙이다. 1NF는 각 컬럼이 원자값을 가져야 하고, 2NF는 부분 함수 종속성을 제거하며(복합 PK에서 일부 컬럼에만 종속된 경우), 3NF는 이행 함수 종속성을 제거한다(PK가 아닌 컬럼이 다른 non-key 컬럼에 종속된 경우). 각 단계를 구체적인 예시로 설명할 수 있어야 한다.

## Level 2: 트레이드오프

### 핵심 질문
- 정규화와 반정규화(Denormalization)의 트레이드오프는 무엇인가? 언제 반정규화를 선택하는가?

### 꼬리 질문
- 정규화된 스키마에서 JOIN이 많아지면 어떤 성능 문제가 발생하는가?
- 반정규화 시 데이터 일관성은 어떻게 유지하는가?

### 예상 이해 수준
정규화는 쓰기 성능과 데이터 정합성에 유리하지만 읽기 시 JOIN이 증가한다. 반정규화는 자주 조회되는 데이터를 한 테이블에 모아 읽기 성능을 높이지만, 데이터 중복으로 인한 불일치 리스크가 생긴다. OLTP(쓰기 많음)는 정규화, OLAP(읽기 많음)는 반정규화가 일반적으로 적합하다. 반정규화된 컬럼은 trigger나 application 로직으로 동기화해야 한다.

## Level 3: 심화

### 핵심 질문
- BCNF(Boyce-Codd Normal Form)와 4NF는 3NF와 어떻게 다른가? Functional dependency란 무엇인가?

### 꼬리 질문
- 3NF를 만족하지만 BCNF를 위반하는 스키마의 예시를 들 수 있는가?
- 4NF에서 말하는 Multivalued dependency란 무엇인가?

### 예상 이해 수준
Functional dependency는 X → Y로 표기하며 X의 값이 Y를 유일하게 결정함을 의미한다. BCNF는 모든 함수 종속성에서 결정자(determinant)가 후보 키여야 한다는 조건으로, 3NF보다 엄격하다. 4NF는 Multivalued dependency를 제거하며, 하나의 컬럼이 독립적으로 여러 값을 가지는 경우(예: 강사-과목, 강사-교재가 독립적으로 다대다)를 분리한다.

## Level 4: 시스템 설계

### 핵심 질문
- 읽기 성능 최적화가 중요한 서비스(예: 소셜 미디어 피드)에서 반정규화 설계를 어떻게 접근할 것인가?

### 꼬리 질문
- 피드에 표시되는 좋아요 수, 댓글 수를 매번 COUNT 집계 없이 어떻게 빠르게 제공할 수 있는가?
- 반정규화된 데이터와 원본 데이터가 일시적으로 불일치하는 상황을 어떻게 허용 가능한 수준으로 관리하는가?

### 예상 이해 수준
소셜 미디어 피드처럼 읽기 빈도가 매우 높은 경우 count 컬럼 캐싱, 요약 테이블(summary table), Materialized view를 활용한다. E-commerce 상품 카탈로그는 카테고리명을 상품 테이블에 비정규화하여 JOIN 없이 목록을 반환한다. 로그 데이터는 애초에 정규화하지 않고 시계열 구조로 저장하는 것이 일반적이다. 데이터 동기화 전략(eventual consistency 허용 여부)을 비즈니스 요구사항과 함께 결정해야 한다.

## 현실 연결
- E-commerce 상품 카탈로그: 상품 테이블에 카테고리명, 브랜드명을 비정규화하면 목록 페이지에서 JOIN 없이 빠르게 조회할 수 있다
- 소셜 미디어 피드: 게시물 테이블에 likes_count, comments_count를 별도 컬럼으로 유지하여 집계 query 없이 피드를 렌더링한다
- 로그 데이터: 클릭 로그, 접속 로그는 정규화 없이 이벤트별 모든 정보를 한 row에 저장하는 flat 구조가 쿼리와 적재에 유리하다
